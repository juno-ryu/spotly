# 공공데이터 API 연동 후보 — 스팟리 확장 로드맵

> 목적: 비서울 지역 데이터 부족 해결 + 스코어링 정밀화
> 작성일: 2026-03-25

---

## 현황 요약

### 현재 연동 데이터소스 (8개)

| # | 데이터소스 | 적용 범위 | 스코어링 기여 |
|---|-----------|---------|-------------|
| 1 | Kakao Places | 전국 | 경쟁 강도 (competition) |
| 2 | 서울 골목상권 | **서울만** | 활력도 (vitality) + 생존율 (survival) |
| 3 | KOSIS 인구 | 전국 | 배후 인구 (population) |
| 4 | 지하철 승하차 | 서울/부산/대구/광주/대전 | 인프라 보너스 (infraBonus) |
| 5 | 버스 정류장 | 전국 | 인프라 보너스 |
| 6 | 학교 DB | 전국 | 인프라 보너스 |
| 7 | 대학교 (Kakao) | 전국 | 인프라 보너스 |
| 8 | 병의원 (Kakao) | 전국 | 인프라 보너스 |

### 핵심 문제

비서울 지역에서 **vitality(활력도)** 와 **survival(생존율)** 지표를 구성할 수 없음.

- 서울: competition×0.25 + vitality×0.35 + population×0.20 + survival×0.20 + infraBonus
- 비서울+P있음+S없음: competition×0.45 + population×0.40 + infraAccess×0.15
- 비서울+P없음: competition×0.75 + infraAccess×0.25

→ 비서울은 경쟁 단일 지표에 과도하게 의존하는 구조.

### 1-1. 소상공인시장진흥공단 상가(상권)정보 API

| 항목 | 내용 |
|------|------|
| **제공처** | 소상공인시장진흥공단 |
| **URL** | https://www.data.go.kr/data/15012005/openapi.do |
| **범위** | **전국** |
| **인증** | `DATA_GO_KR_API_KEY` (기존 키 사용 가능) |
| **갱신** | 분기 |
| **해상도** | 개별 사업소 (위경도) |
| **비용** | 무료 |

**주요 필드**:
- 상호명, 상권업종대/중/소분류명, 표준산업분류명
- 도로명/지번 주소, 경도, 위도
- 건물관리번호, 층정보

**연동 방식**:
```
src/server/data-sources/sbiz/
├── client.ts  — data.go.kr REST 호출 + 반경 조회
└── adapter.ts — 위경도 기반 업종별 점포 수 집계
```

- API 엔드포인트: `GET /storeListInRadius` (반경 내 상가 목록)
- 파라미터: `radius`, `cx`(경도), `cy`(위도), `indsLclsCd`(업종대분류)
- 응답: 상가 목록 (페이징, 1000건/페이지)

**스코어링 반영**:
- **경쟁 강도 정밀화**: 현재 카카오 Places만으로 경쟁업체 수 추정 → 상가정보 DB 병합으로 누락 보완
- 카카오 Places는 "등록된 장소"만 반환 → 상가정보는 사업자등록 기반이라 미등록 업체도 포함
- `competition.ts`에서 `totalCount` 산출 시 두 소스 합집합(중복 제거) 사용
- 가중치 변경 불필요 — 기존 competition 지표의 입력 데이터 품질만 개선

**구현 난이도**: ⭐ (기존 키 사용, 반경 조회 API 제공)

- **매출금액, 매출건수**

**연동 방식**:
```
src/server/data-sources/card-sales/
├── client.ts  — 지자체별 API/파일 수집 통합 어댑터
└── adapter.ts — 행정동코드 기반 매출 지표 산출
```

- 초기 구현: 경기도 API 먼저 → 패턴 확립 후 타 시도 확대
- 파일 데이터(CSV): 월간 배치로 DB 적재 → 조회 시 DB 쿼리

**스코어링 반영**:
- **비서울 vitality 지표 복원** — 이것이 가장 핵심
- 월 매출금액 + 매출건수로 상권 활력도 산출:
  ```
  vitalityScore = normalize(
    monthlyRevenue × 0.5 +
    monthlyTransactions × 0.3 +
    revenueGrowthRate × 0.2
  )
  ```
- 비서울 스코어링 공식 변경:
  - 현재: competition×0.45 + population×0.40 + infraAccess×0.15
  - 개선: competition×0.25 + **vitality×0.30** + population×0.25 + infraAccess×0.20

**구현 난이도**: ⭐⭐⭐ (지자체별 포맷 상이, 통합 어댑터 필요)

### 2-2. 국세청 사업자 현황 (신규/폐업)

| 항목 | 내용 |
|------|------|
| **제공처** | 국세청 |
| **URL** | https://www.data.go.kr/data/15081808/openapi.do |
| **범위** | **전국** (시군구 단위) |
| **인증** | `DATA_GO_KR_API_KEY` |
| **갱신** | 월간~분기 |
| **해상도** | 시군구 단위 |
| **비용** | 무료 |

**주요 필드**:
- 업종별 신규 사업자 수, 폐업 사업자 수
- 시도/시군구 단위 집계

**연동 방식**:
```
src/server/data-sources/nts-business/
├── client.ts  — data.go.kr REST 호출
└── adapter.ts — 시군구별 업종별 신규/폐업 비율 산출
```

**스코어링 반영**:
- **비서울 survival(생존율) 지표 복원**
- 생존율 = 1 - (폐업수 / (기존사업자 + 신규사업자))
  ```
  survivalScore = normalize(survivalRate, {
    min: 0.5,   // 50% 생존 = 0점
    max: 0.95,  // 95% 생존 = 100점
  })
  ```
- 업종별 전국 평균 대비 해당 지역의 생존율 상대 비교
- 비서울 스코어링에 survival 지표 추가 가능:
  - competition×0.25 + vitality×0.25 + population×0.20 + **survival×0.15** + infraAccess×0.15

**구현 난이도**: ⭐⭐ (API 제공, 시군구 단위 한계)

## 비서울 스코어링 개선 로드맵

### 현재 (비서울+P있음+S없음)
```
totalScore = competition×0.45 + population×0.40 + infraAccess×0.15
```

### Phase 1 완료 후 (상가정보 + 주차장 추가)
```
totalScore = competition×0.45 + population×0.40 + infraAccess×0.15
             (competition 정밀도 향상, infraBonus에 주차 추가)
```

### Phase 2 완료 후 (카드소비 + 사업자현황 추가)
```
totalScore = competition×0.25 + vitality×0.25 + population×0.20
             + survival×0.15 + infraAccess×0.15
```

→ **비서울도 서울과 동일한 4+1 지표 구조 달성**

## 기술 구현 패턴

### 신규 데이터소스 추가 표준 절차

```
1. src/server/data-sources/{name}/client.ts 생성
   - API 호출 + 에러 핸들링 + 타임아웃
   - cachedFetch 래핑 (Redis)

2. src/server/data-sources/{name}/adapter.ts 생성
   - 위경도 기반 필터링 (Haversine)
   - 비즈니스 로직 + 타입 export

3. analysis-orchestrator.ts에 병렬 호출 추가
   - Promise.all에 새 fetch 함수 추가
   - catch로 개별 실패 허용 (null 반환)

4. 스코어링 반영 시 박사님(scoring-engine-validator) 검토 필수
   - 가중치 변경은 반드시 사전 승인

5. AI 리포트 반영
   - prompt-builder.ts 데이터 섹션 추가
   - 인사이트 룰 추가 (필요 시)
```
